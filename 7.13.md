# KT6 项目阶段进展简报（7.13）

## 一、阶段结论

目前已完成 **KT6 意图驱动 LUI-GUI 联动的可运行工程原型**，不再是单纯的前端动画演示。系统能够由后端 Runtime 根据用户输入选择业务任务链，驱动左侧拓扑定位、分析、高亮、方案确认、动作执行和结果校验，并保存完整任务轨迹。

当前已验证两类场景：

- 用户网速慢：定位用户及关联 AP，分析同频干扰，人工确认后执行射频调优。
- AP 离线：定位故障 AP，分析 PoE 异常，人工确认后执行端口恢复并校验上线。

## 三、实现思路

整体链路为：

```text
用户输入
-> Intent Agent 解析意图与槽位
-> Router 选择业务 Playbook
-> Runtime 编排诊断步骤
-> 页面感知生成统一 Scene Graph
-> 左侧 GUI 根据事件定位和高亮
-> 右侧生成处置方案并等待人工确认
-> Runtime 校验场景版本、方案权限和资源锁
-> 执行动作 Playbook并校验结果
-> 保存任务轨迹与业务记忆
```

职责划分：Agent 负责理解与诊断，Playbook 沉淀可审计的业务任务链，Runtime 负责可靠编排，感知适配器负责把 DOM、Canvas 或 KT5 拓扑统一转换为 Scene Graph，前端只消费事件并呈现联动结果。

Browser Use 可以作为浏览器会话、DOM 获取、截图和通用点击执行能力使用，但其内置视觉不能单独满足拓扑语义重建。对于只能看到像素的封闭 Canvas，仍需补充专用视觉适配器，完成节点、链路、业务 ID 和跨帧变化识别。

## 四、当前边界与下一步

当前业务指标、拓扑数据和设备操作仍使用 Mock Adapter；Canvas 截图是真实采集，但节点语义主要来自当前页面暴露的渲染器数据，尚未接入纯视觉识别模型。

下一阶段重点：

1. 接入一个真实业务页面，验证 DOM 与可开放 Canvas 数据的感知和执行。
2. 建设 CanvasVisionAdapter，处理纯图片或无法读取内部数据的 Canvas。
3. 将 Mock 业务查询和设备下发替换为真实接口，并补齐鉴权、审计和回滚。
4. 联调 KT5 场景化拓扑生成，验证“感知已有拓扑”和“按意图生成拓扑”两种模式统一运行。

**汇报口径：本阶段完成了 KT6 核心架构和端到端 PoC，已经具备接入真实页面、真实业务接口、专用视觉模型及 KT5 的工程基础，但尚未达到生产交付状态。**
